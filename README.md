# DocsFlow Dashboard

Мини-прототип системы контроля оплат, проектов и закрывающих документов для digital-агентства.

## Что реализовано

- сводка по оплатам, проектам и актам;
- таблица проектов и юридических лиц;
- таблица оплат с этапами работ;
- фильтры по проекту, юрлицу, периоду, статусу акта, этапу и поиску;
- автоматический расчет статусов актов;
- переключение признаков `акт отправлен` и `акт подписан`;
- имитация импорта операций из банковской PDF-выписки;
- сопоставление импортированных оплат с юрлицом, проектом и этапом работ;
- сохранение изменений статусов и комментариев в `localStorage`;
- экспорт текущих оплат в JSON;
- карточка выбранной оплаты с историей документа и комментарием менеджера;
- адаптивная верстка.

## Сущности

### LegalEntity

Клиент или юридическое лицо: название, ИНН, ОГРН, расчетный счет, контактное лицо.

### Project

Проект или направление работ. Связан с юридическим лицом и содержит статус, владельца и дату старта.

### Payment

Оплата по проекту: дата, сумма, назначение платежа, этап услуги, счет и договор.

### Act

Закрывающий документ по оплате: отправлен ли акт, подписан ли акт, даты отправки/подписания и комментарий менеджера.

## Бизнес-логика

Основная логика вынесена в `src/businessLogic.js`.

Статусы актов считаются автоматически:

- `Не отправлен` - акт еще не отправлен и оплата свежая;
- `Ожидает подписи` - акт отправлен, но не подписан;
- `Закрыт` - акт отправлен и подписан;
- `Требует внимания` - оплата старше 10 дней, а акт не отправлен или не подписан.

Фильтрация и расчет итогов также находятся в отдельном слое логики, чтобы UI не смешивался с правилами бизнеса.

## Импорт банковской выписки

В `src/importPipeline.js` показан отдельный слой нормализации данных из банковской выписки.

В прототипе используется мок-выгрузка, имитирующая данные из PDF:

1. строка выписки сопоставляется с юрлицом по ИНН или названию;
2. по юрлицу находится проект;
3. по назначению платежа определяется этап услуги;
4. создается новая оплата;
5. акт для импортированной оплаты получает начальный статус `не отправлен`.

Полноценный PDF-парсер не встроен в браузерный прототип намеренно: в production-версии это должен быть backend/import-сервис, который получает файл, извлекает текст и приводит операции к единому формату `Payment`.

## Допущения

- Данные сейчас моковые и лежат в `src/data.js`.
- Изменение статусов и комментариев сохраняется в `localStorage`.
- Парсинг банковского PDF показан как нормализационный pipeline на мок-операциях. В реальной версии PDF/банк/CRM можно подключить отдельным импорт-сервисом.
- Backend можно добавить без изменения интерфейсной логики: заменить моковые данные REST API и перенести расчет сводок на сервер.

## Развитие до backend-решения

1. Добавить API: `GET /payments`, `PATCH /acts/:id`, `GET /projects`, `GET /legal-entities`.
2. Хранить данные в PostgreSQL или MySQL.
3. Добавить импорт банковской выписки или ручную загрузку файла.
4. Добавить авторизацию и роли менеджера/руководителя.
5. Добавить историю изменений актов и уведомления по просроченным документам.
