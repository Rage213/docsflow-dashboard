# DocsFlow Dashboard

Мини-прототип системы контроля оплат, проектов и закрывающих документов для digital-агентства.

## Что реализовано

- сводка по оплатам, проектам и актам;
- разделы: дашборд, проекты, оплаты, акты, отчеты;
- таблица проектов и юридических лиц;
- таблица оплат с этапами работ;
- фильтры по проекту, юрлицу, периоду, статусу акта, этапу и поиску;
- ручное добавление оплаты с валидацией обязательных полей;
- автоматический расчет статусов актов;
- переключение признаков `акт отправлен` и `акт подписан` через единый слой бизнес-логики;
- скачивание исходной PDF-выписки для проверки демо;
- имитация импорта операций из банковской PDF-выписки;
- валидация строк импорта, учет дублей, нераспознанных и некорректных операций;
- сопоставление импортированных оплат с юрлицом, проектом и этапом работ;
- сохранение изменений статусов и комментариев в `localStorage`;
- экспорт текущих оплат в JSON;
- карточка выбранной оплаты с историей документа и комментарием менеджера;
- экран отчетов с бизнес-правилами и моделью данных;
- адаптивная верстка.

## Модель сущностей

Модель формализована в `src/domainModel.js` и отображается на экране `Отчеты`.

### `legal_entities`

Клиент или юридическое лицо: `id`, `name`, `inn`, `ogrn`, `bankAccount`, `contactPerson`.

### `projects`

Проект или направление работ: `id`, `name`, `legalEntityId`, `status`, `owner`, `startedAt`.

### `payments`

Оплата по проекту: `id`, `projectId`, `legalEntityId`, `paymentDate`, `amount`, `paymentPurpose`, `serviceStage`, `invoiceNumber`, `contractNumber`, `source`.

### `acts`

Закрывающий документ по оплате: `paymentId`, `isSent`, `sentAt`, `isSigned`, `signedAt`, `managerComment`.

## Жесткая бизнес-логика

Основной источник правил — `src/businessLogic.js`.

UI не хранит готовый текстовый статус оплаты. Он меняет только поля акта (`isSent`, `isSigned`, даты), а статус каждый раз вычисляется функцией `getActStatus`.

Статусы актов считаются так:

- `Закрыт` — акт отправлен и подписан;
- `Ожидает подписи` — акт отправлен, но еще не подписан;
- `Не отправлен` — акт не отправлен и оплата еще не просрочена;
- `Требует внимания` — оплате больше 10 дней, а акт не отправлен или не подписан.

Переключение акта идет через `transitionAct`:

- нельзя оставить акт подписанным, если сняли отправку;
- если сразу отметить подпись, система автоматически считает акт отправленным;
- итоговые суммы `Закрыто актами`, `Незакрыто`, `Ждут подписи`, `Требуют внимания` пересчитываются из текущего состояния оплат.

## Импорт банковской выписки

В `src/importPipeline.js` показан отдельный слой нормализации данных из банковской выписки.

В прототипе используются строки из предоставленной PDF-выписки `bank_statement_project_data_clean.pdf`, заранее приведенные к JSON-формату для браузерного демо. Сам PDF лежит в `public/` и доступен на сайте через кнопку `Скачать PDF`.

Pipeline делает следующее:

1. проверяет обязательные поля операции: id, дату, сумму, плательщика, назначение платежа;
2. отделяет некорректные строки в `invalidRows`;
3. сопоставляет плательщика с юрлицом по ИНН или названию;
4. если юрлицо или проект не найдены, кладет строку в `unresolvedRows`;
5. по назначению платежа определяет этап услуги;
6. создает новую оплату;
7. начальный акт для импортированной оплаты получает статус `не отправлен`.

Полноценный PDF-парсер не встроен в браузерный прототип намеренно: в production-версии это должен быть backend/import-сервис, который получает файл, извлекает текст и приводит операции к единому формату `Payment`.

## Архитектура

- `src/App.jsx` — координатор состояния, фильтров, импорта и действий пользователя;
- `src/businessLogic.js` — расчет статусов, итогов, фильтрация, переходы состояния акта;
- `src/importPipeline.js` — нормализация и валидация строк банковской выписки;
- `src/domainModel.js` — формализованная модель сущностей;
- `src/components/` — презентационные компоненты интерфейса;
- `src/data.js` — демо-данные.

## Допущения

- Данные сейчас моковые и лежат в `src/data.js`.
- Изменение статусов и комментариев сохраняется в `localStorage`.
- Backend можно добавить без изменения большей части UI: заменить моковые данные REST API и перенести `businessLogic.js` на сервер.
- В реальной версии нужна история изменений актов, авторизация и роли менеджера/руководителя.

## Развитие до backend-решения

1. Добавить API: `GET /payments`, `PATCH /acts/:id`, `GET /projects`, `GET /legal-entities`.
2. Хранить данные в PostgreSQL или MySQL.
3. Перенести `getActStatus`, `transitionAct`, `calculateSummary` на backend как single source of truth.
4. Добавить импорт банковской выписки или ручную загрузку файла.
5. Добавить авторизацию и роли менеджера/руководителя.
6. Добавить историю изменений актов и уведомления по просроченным документам.
